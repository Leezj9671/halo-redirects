# Redirects Plugin Roadmap

## 背景

当前插件定位是一个轻量级的 Halo 2.x 重定向工具，核心能力是：

- 在插件设置页维护规则
- 支持批量导入
- 支持 301 / 302
- 支持保留查询参数
- 规则变更后自动热加载

当前还需要额外记录一个上游边界：

- 插件自带的 Halo Console 前端扩展如果加载失败或注册异常，可能会连带影响其他插件的左侧菜单显示，例如“应用市场”。
- 这类影响范围不应跨插件扩散，理论上应该由 Halo Console 在加载层做隔离和容错。
- 因此在 Halo 上游没有明确兜住这类异常之前，`redirects` 不应优先恢复自定义控制台页面，避免把非核心功能变成后台稳定性风险。

现有架构偏“设置页驱动”：

- 规则保存在插件设置中
- 运行期使用内存注册表匹配
- 请求入口是一个 `WebFilter`

这套结构适合“静态规则维护”，但不适合直接承载以下能力：

- 高频写入日志
- 独立列表页和交互按钮
- 实时测试与统计面板
- 基于内容变更事件的自动建议

因此，后续功能规划要先区分两条路线：

- 小步快跑版：尽量复用现有设置结构，先补高价值护栏功能
- 产品升级版：引入独立管理页和独立数据模型，做真正的“留存型功能”

## 目标

做出 3-4 个用户能明显感知、并且足以拉开差距的功能，提升：

- 留存率
- 好评率
- 使用频次
- 对 Halo 用户的迁移价值

不追求一次性做全，而是按“高感知 + 可控复杂度”推进。

## 功能优先级结论

建议优先顺序如下：

1. 404 错误监控 + 一键创建重定向
2. 测试 / 预览功能
3. 每条规则命中次数 + 最后命中时间
4. 重定向链检测 / 循环警告

以下功能建议后置：

- 自动创建重定向（内容变更联动）
- 批量操作（勾选、批量启停、批量删除、筛选）
- 完整统计仪表盘（趋势图、Top 10、首页面板）

## 功能判断

### 1. 404 错误监控 + 一键创建重定向

结论：非常值得做，属于第一优先级。

价值：

- 直接命中用户最痛的场景：发现 404 并修复
- 使用闭环非常清晰：发现问题 -> 创建规则 -> 解决问题
- 这是最容易让用户觉得“这个插件真有用”的能力

建议形态：

- 增加独立的“404 日志”页签或管理页
- 记录 URL、首次时间、最后时间、次数、最近访问来源（可选）、IP（可选）
- 每条记录旁提供“创建重定向”按钮
- 点击后自动预填 `source`

实现前提：

- 需要独立持久化存储，不能只靠 settings
- 需要独立接口和管理页，不适合继续塞到设置表单里

注意点：

- 过滤掉 `/api`、`/console`、静态资源等噪音请求
- 控制日志体量，支持只保留最近 N 条或按天清理
- IP 采集涉及隐私，建议作为可选项

### 2. 测试 / 预览功能

结论：非常值得做，属于低成本高感知功能。

价值：

- 降低误配置风险
- 减少用户对规则是否生效的不确定感
- 为后续 regex、自动生成规则、批量导入提供安全护栏

建议形态：

- 在新增 / 编辑规则时提供“测试 URL”
- 输入后实时返回：
  - 是否命中
  - 命中的规则
  - 最终跳转地址
  - 状态码
  - 未命中原因（可选）

实现前提：

- 新增一个只读测试接口
- 复用当前实际匹配逻辑，避免“测试结果”和线上行为不一致

这项功能即使在不改大架构的情况下也可以先做。

### 3. 重定向命中统计（先做轻量版）

结论：值得做，但应先做轻量版，不要一上来做完整仪表盘。

建议第一阶段只做：

- 每条规则命中次数
- 最后命中时间

价值：

- 用户能知道哪些规则真正在被访问
- 能辅助判断哪些旧链接还活跃
- 比“首页统计大盘”更贴近日常管理

为什么不建议先做完整仪表盘：

- 趋势图、Top 10、按天聚合会显著增加数据模型和前端工作量
- 对早期版本的实际价值，不如“单条规则统计”直接

实现前提：

- 规则需要稳定 ID，不能只用 `source path` 当唯一标识
- 统计数据需要独立存储

建议：

- 先做规则维度的累计计数
- 后续再扩展日维度趋势统计

### 4. 重定向链检测 / 循环警告

结论：应该做，属于基础安全能力。

价值：

- 避免出现无限跳转
- 避免用户误配置导致体验或 SEO 问题
- 能显著提高“插件靠谱”的感知

建议检测内容：

- 自循环：`A -> A`
- 直接环：`A -> B -> A`
- 间接环：`A -> B -> C -> A`
- 可选：长链提醒，如 `A -> B -> C`

实现建议：

- 在规则加载时建立站内路径跳转图
- 检测环路
- 保存失败时直接阻止，或至少给出强警告

这项功能可以较早落地，且不依赖完整管理页。

### 5. 自动创建重定向（Halo 专属能力）

结论：很有差异化，但建议第二阶段再做。

价值：

- 这是最具 Halo 生态特色的功能
- 对博客迁移、改 slug、调整 permalink 结构很有帮助
- 如果做稳，会成为插件的“记忆点”

为什么不建议第一批就做：

- 会把插件从“规则工具”升级为“内容生命周期联动工具”
- 会涉及文章、分类、链接结构变更事件
- 边界很多：草稿、批量修改、重复规则、误判、回滚

建议形态：

- 监听内容相关事件
- 当检测到旧路径变更为新路径时，不直接写入规则
- 先生成“待确认建议”
- 由用户确认后一键创建 301

原则：

- 默认“建议模式”，不要静默自动创建

### 6. 批量操作

结论：有价值，但不应早于前面四项。

当前判断：

- 插件已经有“批量导入”，所以“批量新增”并不完全缺失
- 真正昂贵的是列表选择、批量启停、批量删除、导出和筛选

什么时候值得做：

- 当规则数量上升到几十条或上百条
- 当用户主要场景转向迁移站、内容重构、集中治理时

实现前提：

- 需要真正的列表页，而不是设置页 repeater

### 7. 完整统计仪表盘

结论：可以做，但排在规则统计之后。

适合在以下条件满足后再做：

- 已有稳定规则 ID
- 已有命中统计
- 已有独立管理页

建议包含：

- 今日命中
- 总命中
- Top 10 活跃规则
- 最近 7 / 30 天趋势

如果前期资源有限，这项可以暂缓，不影响核心体验。

## 推荐实施路线

### 路线 A：保守升级（尽量复用现有架构）

适合目标：

- 先用最小成本提升可靠性和易用性
- 暂时不引入完整自定义管理页

建议顺序：

1. 重定向链检测 / 循环警告
2. 测试 / 预览接口
3. 轻量命中统计（如果能接受简单存储）

优点：

- 改动小
- 发布快
- 风险低

缺点：

- 404 日志和一键创建体验会受限
- 后续仍可能需要重构

### 路线 B：产品升级（推荐）

适合目标：

- 做出一个明显高于普通重定向插件的版本
- 强化长期留存和差异化

第一阶段先做基础设施：

- 独立规则数据模型（至少具备稳定 ID）
- 独立日志 / 统计存储
- 自定义管理页
- 独立 API

在此基础上，第一批功能建议：

1. 404 错误监控 + 一键创建重定向
2. 测试 / 预览功能
3. 规则命中次数 + 最后命中时间
4. 重定向链 / 循环检测

第二批再做：

1. 自动创建重定向（建议模式）
2. 批量操作
3. 完整统计仪表盘

优点：

- 用户感知强
- 结构更合理
- 后续扩展成本更低

缺点：

- 首次改造成本明显更高

## 技术架构建议

建议把后续架构从“设置页规则”升级为“两层结构”：

### 核心匹配层

负责：

- 规则加载
- 规则匹配
- 状态码返回
- 查询参数保留
- 循环检测

要求：

- 保持足够轻量
- 继续以内存快照承载在线匹配，保证性能

### 管理与数据层

负责：

- 规则 CRUD
- 404 日志
- 命中统计
- 测试接口
- 自动建议

要求：

- 持久化存储
- 独立 API
- 可扩展为自定义管理页

核心原则：

- 在线请求路径继续走内存快照，避免每次请求都查库
- 后台管理能力走独立数据层，避免把 settings 当数据库用

## 建议的里程碑

### Milestone 1：稳住核心能力

目标：

- 提高规则安全性和可验证性

包含：

- 循环检测
- 测试 / 预览接口
- 更清晰的规则校验提示

### Milestone 2：补上高感知留存功能

目标：

- 让用户能快速发现并修复 404

包含：

- 404 日志
- 一键创建重定向
- 基础日志清理策略

### Milestone 3：做数据化管理

目标：

- 让规则从“能用”升级到“可运营”

包含：

- 每条规则命中次数
- 最后命中时间
- 后续扩展趋势图所需的数据基础

### Milestone 4：做 Halo 专属差异化

目标：

- 建立真正的生态特性壁垒

包含：

- 监听内容链接变更
- 生成自动重定向建议
- 用户确认后写入规则

## 当前建议

如果只做一版就想明显提升口碑，推荐直接以“路线 B”推进，但控制首批范围：

首批只做四项：

1. 404 错误监控 + 一键创建重定向
2. 测试 / 预览功能
3. 每条规则命中次数 + 最后命中时间
4. 重定向链 / 循环检测

这样既有用户能立刻感知的亮点，也不会因为范围过大拖慢交付。

## 暂不建议做的事

以下内容不建议现在优先投入：

- 为了“看起来丰富”而先做复杂仪表盘
- 在没有稳定规则 ID 的情况下直接做统计体系
- 在没有确认流程的情况下直接做自动重定向写入
- 继续把日志、统计、操作状态都塞进插件 settings

这些做法短期看似快，后期维护成本会很高。
